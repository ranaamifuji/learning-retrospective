初心者FAQ：エージェントに確認して分かったこと＋

実際に迷った内容を「問 → 回答 → 理由」の順で整理しています。確認手順を上から1つずつ試してください。画面キャプチャは、文章だけではクリック場所や表示を判別しにくい項目に限って追加します。

問1：チャットを始めるたびに「バイブコーディングマスター」フォルダを指定する必要がありますか？

回答

毎回チャットへフォルダ名を書く必要はありません。Cursorで作業フォルダとして一度開き、そのウィンドウ内でチャットします。

理由

エージェントは、Cursorで現在開いているワークスペースを基準にファイルを探します。フォルダを開かずに始めると、対象ファイルや保存先を正しく判断できません。

確認手順

  1. Cursorの「File → Open Folder」で バイブコーディングマスター を開く
  2. 左側に docs、projects、学習の振り返り が見えることを確認する
  3. 同じウィンドウのチャットで対象ファイルを伝える

成功の目印

エージェントが対象ファイルを追加説明なしで読み取れ、意図したフォルダ内だけを扱える。

専門用語：ワークスペース＝Cursorで現在開いている作業フォルダ。

関連プロンプト

現在開いているワークスペースと対象フォルダを確認して。まだファイルは変更しないで。
問2：白い大きなチャット画面から、黒いエディタ画面に変わったのはなぜですか？

回答

フォルダを開いたことや、チャット中心画面から通常のIDEレイアウトへ切り替わったことが主な理由です。不具合とは限りません。

理由

Cursorには、チャットを大きく表示する画面と、ファイル一覧・エディタ・チャットを同時に表示する画面があります。また、テーマ設定によって背景色も変わります。

確認手順

  1. 左側にファイル一覧が表示されているか確認する
  2. 開いているフォルダ名が目的のワークスペースか確認する
  3. 作業ファイルとチャットを一緒に使う場合は、そのまま通常画面で進める

成功の目印

左のファイルを選ぶと中央で開き、同じ画面内のチャットからそのファイルについて質問できる。

専門用語：IDE＝ファイル編集、ターミナル、AIチャットなどを一つにまとめた開発環境。

問3：GitをインストールしてCursorを開き直したのに、gitコマンドが見つからないのはなぜですか？

回答

Git本体が入っていても、起動中のCursorが古いPATHを引き継いでいることがあります。Cursorを完全終了して再起動し、それでも直らなければWindowsをサインアウトまたは再起動します。

理由

Gitのインストーラーは通常、Gitの場所をWindowsのPATHへ追加します。ただし、すでに動いているアプリには新しい環境変数が自動反映されない場合があります。

確認手順

  1. Cursorのすべてのウィンドウを閉じ、バックグラウンドにも残っていない状態にする
  2. Cursorを起動し直して、新しいターミナルを開く
  3. git --version を実行する
  4. 認識されなければWindowsをサインアウトまたは再起動する

成功の目印

git version 2.55.0.windows.3 のようにバージョン番号が表示される。

専門用語：PATH＝コマンドの実体をWindowsが探すフォルダ一覧。

関連プロンプト

Git本体の有無、インストール先、WindowsのPATH、現在のCursorが読み込んでいるPATHを確認して。設定はまだ変更しないで。
問4：エディタに内容が見えるのに、GitHubにファイルがないのはなぜですか？

回答

画面に表示されているだけで、ファイルとして保存されていない可能性があります。まず Ctrl + S で保存します。

理由

エディタ上の未保存内容、PCに保存されたファイル、Gitの履歴、GitHub上のファイルはそれぞれ別の状態です。保存前の内容はGitから見えません。

確認手順

  1. Ctrl + S で保存する
  2. 対象フォルダで git status を実行する
  3. 変更が表示されたら git add、git commit、git push の順で進める

成功の目印

git status に対象ファイルが表示され、push後にGitHub上でも確認できる。

専門用語：保存＝編集中の内容をPC上のファイルへ書き込むこと。push＝記録済みの変更をGitHubへ送ること。

問5：変更したのに、git commitで「nothing to commit」と表示されるのはなぜですか？

回答

git add が済んでいないか、変更が保存されていない可能性があります。git status で状態を確認します。

理由

commitは、変更されたすべてのファイルではなく、先にaddで選ばれた変更を記録します。

確認手順

  1. ファイルを保存する
  2. git status で対象ファイルを確認する
  3. git add ファイル名 を実行する
  4. もう一度commitする

成功の目印

commit結果に変更されたファイル数や追加・削除行数が表示される。

専門用語：add＝次のcommitへ含める変更を選ぶこと。commit＝選んだ変更を履歴へ記録すること。

問6：cdを実行したら「パスが見つからない」と表示されるのはなぜですか？

回答

すでに目的のフォルダ内にいるのに、同じ移動を繰り返している可能性があります。先に現在地を確認します。

理由

相対パスの cd projects/01-profile-site は「今いる場所から、その下へ移動する」という命令です。同じ場所で繰り返すと、存在しない二重のパスを探します。

確認手順

  1. PowerShellの行頭に表示される現在地を見る
  2. pwd を実行して現在地を確認する
  3. 目的地にすでにいる場合は、cdを実行せず次の操作へ進む

成功の目印

現在地の末尾が目的のフォルダ名になり、次のコマンドが正しい対象で動く。

専門用語：相対パス＝現在地を基準にした行き先。絶対パス＝ドライブ名から始まる場所全体の住所。

問7：git pushに成功したのに、公開ページが変わらないのはなぜですか？

回答

GitHub Pagesの反映待ちか、ブラウザに古い表示が残っている可能性があります。push成功を確認し、数分待って強制再読み込みします。

理由

pushはGitHubへ履歴を送る処理です。その後、GitHub Pagesが公開用ページを作り直すため、表示まで時間差があります。

確認手順

  1. push結果に main -> main があるか確認する
  2. GitHub上の対象ファイルが更新されているか確認する
  3. 1〜数分待つ
  4. Ctrl + F5 で強制再読み込みする

成功の目印

GitHub上のファイルと公開ページの両方に新しい内容が表示される。

専門用語：キャッシュ＝表示を速くするためブラウザが一時保存した古いデータ。

問8：BRIEFの「画面」と「操作」は、どう分ければよいですか？

回答

画面には「見える物」を名詞で、操作には「何をすると、どうなるか」を動詞で書きます。

理由

部品と動作を分けると、作る対象と期待する結果が明確になり、エージェントの実装ずれを減らせます。

確認手順

  1. 写真に写るものは「画面」へ書く
  2. 指やマウスを動かして起きることは「操作」へ書く
  3. 一つの文に混ざっていたら、部品と動作の2文に分ける

成功の目印

「画面：追加ボタン」「操作：追加ボタンを押すと一覧に商品が増える」のように対応している。

専門用語：BRIEF＝実装前に目的、画面、操作、完了条件、やらないことを決める短い仕様書。

関連プロンプト

BRIEFの「画面」と「操作」を分けて整理して。画面は見える物、操作はやること→結果。実装はまだしないで。

公開の基本動作（Gitの流れ）＋

編集 → Ctrl+S → git status → git add → git commit → git push

操作意味
Ctrl + S編集中の内容をPCのファイルへ保存
git status変更されたファイルと現在地を確認
git add index.html次の記録に含めるファイルを登録
git commit -m "理由"変更を理由付きで履歴に記録
git push記録をGitHubへ送信

公開ページはpush後すぐでなく、反映に1〜数分かかることがある。古い表示なら Ctrl + F5 で強制再読み込み。

つまずき事例（症状 → 原因 → 解決）＋

1. 画面に見えるのにGitHubにファイルが無い

症状: git status は変更なし、ls に README しか出ない

原因: エディタの内容が保存されていなかった

解決: Ctrl+S で保存、またはGitHubのUpload filesで直接アップロード

補足: 「見えている」と「保存済み」は別

2. commitしても nothing to commit

症状: modified と出るのに記録されない

原因: git add をしていなかった

解決: add → commit → push の順で実行

補足: add は「今回の記録に入れるものを選ぶ」工程

3. cd でパスが見つからない

症状: パスに 01-profile-site が二重に続く

原因: すでにそのフォルダにいるのに同じ cd を繰り返した

解決: ターミナル左端の現在地を確認してから移動

4. pushしたのにページが変わらない

原因候補: 反映待ち／ブラウザのキャッシュ／push前

解決: push成功を確認 → 数分待つ → Ctrl+F5

目印: push結果に main -> main が出る

5. Gitのセットアップ項目が多くて迷う

方針: 基本は推奨・初期値でよい

重要: PATHは3rd-party許可、認証はGit Credential Manager

補足: VS Code Insidersは今回不要。Cursorだけで進められる

6. BRIEFの「画面」と「操作」のどちらに書くか迷う

症状: やりたいことや項目名を、どちらに書けばよいか判断できない

原因: 「見える物」と「やること・結果」が同じ文に混ざる

解決: 画面＝名詞／部品、操作＝動詞／手順→結果、で振り分ける

補足: 迷ったら「写真に写るか／指を動かすか」で考える

Cursorの使い方＋

  1. Cursorで作業フォルダを開く
  2. 先にBRIEFを書く（目的と完了条件）
  3. Cursorに「BRIEFを読み実装。やらないことは作らない」と依頼
  4. 自分でブラウザ表示を確認
  5. GitHubで公開リポジトリを作る
  6. 最初はUpload filesでも index.html を置く
  7. GitHub Pagesを main と / (root) で有効化
  8. 慣れたらGitを導入し add → commit → push へ
  9. 完成後は振り返りを書く

AI利用量を抑える考え方＋

  • 新規設計・初回セットアップ・原因不明のエラーはAIに任せる価値が高い
  • 文言やURLの1箇所変更は自分で編集できる
  • 自分で直したあとは「commit・push・確認だけ」と範囲を絞って依頼
  • Python等の自動化は、同じ処理を何度も繰り返す段階で効く

セキュリティ設定で学んだこと＋

実施したこと

  • 今後のコミット用メールをGitHubのnoreplyアドレスへ変更
  • メールアドレスを非公開にする設定をOn
  • 個人メールを含むpushをGitHub側でブロックする設定をOn
  • 認証アプリによる2段階認証（2FA）を有効化
  • リカバリーコードを安全な場所に保存
  • 画面共有に写ったコードを再発行し、古いコードを無効化

覚えておくこと

項目意味
noreplyメール個人メールを公開せず、GitHubアカウントとコミットを結びつける
2段階認証パスワード以外の確認を追加し、乗っ取りを防ぎやすくする
リカバリーコード認証端末を失ったときに使う1回限りの復旧手段
パスワード、認証コード、リカバリーコード、APIキーは、チャット・スクリーンショット・GitHubに載せない。写った場合は再発行して古いものを無効化する。

BRIEFから始める

（対象フォルダ）のBRIEF.mdを読んで実装して。完了条件を満たし、やらないことは作らない。最後にブラウザで確認して。

質問だけする

実装・ファイル変更はしないで質問だけ。〇〇について初心者向けに、今のプロジェクトを例に説明して。

公開前の安全確認

公開・push前に、メールアドレス、パスワード、APIキー、認証コード、個人情報が含まれていないか確認して。問題があればpushせず報告して。

BRIEFの画面と操作を整理する

BRIEFの「画面」と「操作」を分けて整理して。画面は見える物、操作はやること→結果。実装はまだしないで。

基本版

今日の学びを蓄積して。今日実施したこと、学んだ用語、つまずいてエージェントへ確認した内容（問・回答・理由・確認手順・成功の目印・専門用語）、効いた指示、初心者向け補足、次回の1手を、日付ごとの学習ログと振り返りサイトの初心者FAQに追記して。秘密情報や個人情報は記録・公開しないで。GitHubへのpushはしないで。

短縮版

今日の学びを、秘密情報を除いて日次ログと振り返りサイトに蓄積して。つまずきは初心者FAQ形式でも残して。GitHubには上げないで。
